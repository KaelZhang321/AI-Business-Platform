# 优化方案08：P6技术栈独立性应对方案

> 解决问题：P6是唯一Vue项目，组件/状态管理不能与React产品线复用
> 优先级：P1（高优先级，Q1-Q2评估决策）
> 工作量：评估1天 + 方案确认后按选择执行
> 影响范围：P6 AI企微侧边栏

---

## 一、问题分析

### 1.1 当前技术栈对比

| 维度 | P1-P5/P7/P8（7个产品） | P6（1个产品） |
|------|----------------------|--------------|
| 前端框架 | React 18 | Vue 3 |
| UI组件库 | Ant Design 5 | Vant 4 |
| 状态管理 | Redux / Zustand | Pinia |
| 类型系统 | TypeScript | TypeScript |
| 构建工具 | Vite | Vite |
| BFF层 | 无（直连Spring Boot） | Node.js + NestJS |
| 运行环境 | 浏览器（PC端） | 企微侧边栏（移动H5） |

### 1.2 痛点量化

| 痛点 | 影响 | 严重度 |
|------|------|--------|
| 组件不能复用 | AI Copilot、360视图等核心组件需用Vue重写 | 高 |
| 状态管理不统一 | Redux逻辑不能迁移，需用Pinia重写 | 中 |
| BFF额外维护 | 多一层Node.js需要运维，增加复杂度 | 中 |
| 人员技能分散 | 5人团队需同时维护React+Vue+Node.js | 高 |
| 共享包困难 | 工具函数/类型定义需Framework-agnostic | 低 |

### 1.3 P6选择Vue的历史原因

1. **企微生态**：企微开发者社区Vue示例更多，Vant 4是企微生态主流移动端组件库
2. **H5轻量性**：Vue 3 + Vant打包体积更小（~150KB vs React+AntD Mobile ~250KB）
3. **开发速度**：Vue SFC单文件组件开发效率高，适合侧边栏场景

## 二、方案评估

### 2.1 三个候选方案

| 方案 | 描述 | 工作量 | 风险 |
|------|------|--------|------|
| **A：保持Vue，抽取共享层** | 维持现状，通过共享npm包减少重复 | 低（2周） | 长期维护成本高 |
| **B：迁移至React + AntD Mobile** | P6前端重写为React | 高（4-6周） | 迁移期功能冻结 |
| **C：Web Components桥接** | 核心组件封装为Web Components跨框架复用 | 中（3-4周） | 技术复杂度高 |

### 2.2 详细评估

#### 方案A：保持Vue + 共享npm包（推荐 ✅）

```
┌─────────────────────────────────────────────┐
│              共享层架构                       │
│                                              │
│  @lizkal/shared-types      TypeScript类型     │
│  @lizkal/shared-utils      工具函数           │
│  @lizkal/shared-api        API客户端          │
│  @lizkal/shared-constants  常量/枚举          │
│                                              │
│         ┌────────────┐  ┌────────────┐       │
│         │  React Apps │  │  Vue App   │       │
│         │ P1-P5/P7/P8│  │    P6      │       │
│         │ Ant Design 5│  │  Vant 4    │       │
│         └──────┬─────┘  └──────┬─────┘       │
│                └───────┬───────┘              │
│                        ▼                      │
│             @lizkal/shared-*                  │
│           (Framework-agnostic)                │
└─────────────────────────────────────────────┘
```

**共享包设计**：

```typescript
// @lizkal/shared-types - 跨框架类型定义
export interface PatientProfile {
  id: string;
  name: string;
  gender: 'M' | 'F';
  age: number;
  healthTags: string[];
  riskLevel: 'LOW' | 'MEDIUM' | 'HIGH';
}

export interface AIRecommendation {
  id: string;
  type: 'SCRIPT' | 'PRODUCT' | 'SERVICE';
  content: string;
  confidence: number;
  sources: RAGSource[];
}

// @lizkal/shared-api - 统一API客户端
export class LizkalAPIClient {
  constructor(private baseURL: string, private token: string) {}

  async getPatientProfile(patientId: string): Promise<PatientProfile> {
    return this.get(`/api/v1/patients/${patientId}/profile`);
  }

  async getAIRecommendation(context: AIContext): Promise<AIRecommendation> {
    return this.post('/api/v1/ai/recommend', context);
  }
}

// @lizkal/shared-utils - 纯函数工具
export function maskPhone(phone: string): string {
  return phone.replace(/(\d{3})\d{4}(\d{4})/, '$1****$2');
}

export function formatHealthRisk(level: string): { text: string; color: string } {
  const map = { LOW: { text: '低风险', color: '#52c41a' }, ... };
  return map[level];
}
```

**优势**：
- 零迁移风险，不影响P6开发进度
- 共享层估计覆盖30-40%的代码逻辑
- TypeScript类型+API客户端+工具函数完全复用
- 2周即可完成共享包抽取

**劣势**：
- UI组件层仍需各自维护
- 长期需同时维护React+Vue两套UI逻辑

---

#### 方案B：迁移至React + AntD Mobile

**可行性评估**：

| 维度 | 评估 |
|------|------|
| AntD Mobile 适配企微 | ✅ 可行，AntD Mobile 5.x支持H5 |
| 企微JS-SDK | ✅ 框架无关，React/Vue均可调用 |
| Vant 4专有组件 | ⚠️ ActionSheet/SwipeCell等需找AntD Mobile替代 |
| 打包体积 | ⚠️ React+AntD Mobile比Vue+Vant大约50-100KB |
| 首屏性能 | ⚠️ P6要求≤1.5s首屏，需验证 |
| 企微社区支持 | ⚠️ React在企微生态示例较少 |

**迁移工作量估算**：

| 模块 | Vue组件数 | 迁移工作量 | 风险 |
|------|----------|-----------|------|
| 客户画像面板 | ~8个 | 3天 | 低 |
| AI话术推荐 | ~5个 | 2天 | 低 |
| 快捷操作 | ~6个 | 2天 | 中（企微SDK适配） |
| 通话助手 | ~4个 | 2天 | 中 |
| 素材中心 | ~5个 | 2天 | 低 |
| 数据看板 | ~4个 | 2天 | 低 |
| BFF移除+直连 | — | 3天 | 高（接口重构） |
| 联调测试 | — | 5天 | 中 |
| **合计** | ~32个 | ~21天 | — |

---

#### 方案C：Web Components桥接

**技术方案**：
```
核心AI组件 → Lit/Stencil封装 → Web Components → React/Vue均可使用
```

**评估结论**：**不推荐**
- AI Copilot等组件与框架状态深度绑定，抽离成本高
- Web Components在企微webview兼容性存在风险
- 增加了第三种技术栈（Lit/Stencil），与简化目标矛盾

## 三、推荐决策

### 3.1 决策矩阵

| 评估维度 | 权重 | 方案A（共享层） | 方案B（迁移React） | 方案C（Web Components） |
|---------|------|---------------|-------------------|----------------------|
| 实施风险 | 30% | 5分 | 3分 | 2分 |
| 长期维护成本 | 25% | 3分 | 5分 | 2分 |
| 实施工作量 | 20% | 5分 | 2分 | 3分 |
| 团队技能统一 | 15% | 2分 | 5分 | 1分 |
| 性能影响 | 10% | 5分 | 3分 | 4分 |
| **加权总分** | **100%** | **4.05** | **3.55** | **2.25** |

### 3.2 最终推荐

```
┌─────────────────────────────────────────────┐
│           P6 技术栈决策                      │
│                                              │
│  短期（Q1-Q2）：方案A ✅                     │
│    → 保持Vue 3 + Vant 4                     │
│    → 抽取@lizkal/shared-*共享npm包           │
│    → 移除NestJS BFF，改为直连Spring Boot     │
│                                              │
│  中期（Q3评估）：                             │
│    → 评估P6功能复杂度增长趋势                 │
│    → 若P6功能持续扩展，Q4启动方案B迁移        │
│    → 若P6功能稳定，继续方案A                  │
│                                              │
│  长期：                                      │
│    → P6侧边栏定位为"轻量触点"                │
│    → 复杂功能引导至React主应用完成             │
│    → 侧边栏保持轻量不膨胀                     │
└─────────────────────────────────────────────┘
```

### 3.3 BFF层优化（无论选A或B都执行）

当前P6有一层NestJS BFF，增加了运维复杂度。建议移除：

```
当前：P6 Vue → NestJS BFF → Spring Boot → AI Service
优化：P6 Vue → Spring Boot（P8网关） → AI Service
```

**理由**：
1. Spring Cloud Gateway已提供统一网关能力
2. BFF的聚合逻辑可在Spring Boot层完成
3. 减少一个Node.js服务的运维负担
4. 5人团队不建议维护3种后端语言（Java+Python+Node.js）
