### 1.2 架构设计原则

| 原则 | 说明 | 落地方式 |
|------|------|---------|
| **统一底座** | 所有AI能力由P8技术中台统一提供 | 模型调用、知识检索、工作流编排均通过中台API |
| **松耦合** | 各应用层产品互不依赖，仅依赖中台 | 通过API网关解耦，独立部署独立迭代 |
| **可复用** | 中台能力组件化，各工作台按需组合 | 向量RAG、GraphRAG融合检索、LLM、工作流、Agent框架（LangChain+LangGraph）等能力标准化封装 |
| **渐进式交付** | 从核心到扩展，从MVP到全功能 | 每个产品7-8个版本，2周一个Sprint |
| **数据驱动** | 所有AI交互数据可追踪、可评测、可优化 | 统一监控、A/B测试、效果评估闭环 |

### 1.3 技术栈统一规范

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端（P1-P5/P7/P8）** | React 18 + TypeScript + Ant Design 5.x | 统一组件库，五栏布局标准。P6特例采用Vue 3 + Vant 4（详见优化方案08） |
| **后端** | Spring Boot 3.x + MySQL 8.0 + Redis 7.x | 微服务架构，统一数据层 |
| **AI层** | DeepSeek/通义千问 + LangChain + LangGraph + Milvus | 多模型路由，RAG+GraphRAG知识检索（详见补充方案02、优化方案11） |
| **消息队列** | RabbitMQ | 异步任务处理 |
| **可视化** | ECharts + React-Grid-Layout | 数据看板与拖拽布局 |
| **企微集成** | 企业微信开放平台SDK | 侧边栏应用开发 |
| **语音处理** | 阿里云语音ASR/TTS | 语音转文字、语音交互 |
| **认证授权** | Keycloak 24.x + OAuth2/OIDC + JWT | 统一SSO认证，跨产品单点登录（详见补充方案06） |
| **缓存** | Caffeine（L1本地）+ Redis 7.x（L2分布式）| 三级缓存体系，语义缓存降成本40%（详见优化方案09） |

### 1.4 AI+Vibe Coding交付策略

> 依据《AI工作年度规划_2026_最终版》和《02_执行计划/05_月度工作计划_详细到人天.md》的流程基线，结合最新人力约束与Vibe Coding开发要求重新梳理。

- **五人制跨职能战队**：仅配置1名产品、1名AI架构师、1名Java工程师、1名前端工程师、1名算法工程师。测试与运维职责由架构师与产品在Sprint验收中共担，确保不突破既定人力边界。
- **Vibe Coding工作流**：全部开发任务在Vibe Coding环境中完成，借助AI结对编码、自动补全与实时代码审查能力，形成“需求→Prompt→实现→自测→回放”闭环。每日提交需保留Prompt与生成记录，方便复盘与合规存档。
- **知识底座同步**：将《AI业务调研.md》《AI部门工作职责说明.md》等原始资料预先纳入知识库，Vibe Coding中的Agent可直接检索原始需求，保障“不凭空杜撰”原则。
- **质量与安全门禁**：每个Vibe会话需附带Checklist（需求映射、数据合规、性能基线），由架构师在Merge前逐条确认；结构化日志同步到P8监控模块，用于版本后评估。
