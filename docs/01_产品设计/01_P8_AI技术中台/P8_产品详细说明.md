## 一、产品概述

### 1.1 产品定位

P8 AI技术中台是丽滋卡尔企业级AI能力底座与统一基础设施平台，承担为所有上层AI应用（P1-P7）提供标准化、可复用、可治理的AI能力服务的使命。

### 1.2 产品使命

**"统一底座、能力资产化、降本增效、安全合规"**

- **统一底座**：避免各业务线重复建设AI基础设施，集中管理模型、知识、算力资源
- **能力资产化**：将AI能力沉淀为企业资产（Prompt模板、知识库、工作流、智能体），降低人员流失风险
- **降本增效**：通过统一模型路由、缓存策略、成本治理，降低模型调用成本30%+
- **安全合规**：统一鉴权、全链路审计、数据脱敏，满足医疗健康行业合规要求

### 1.3 目标用户

| 用户角色 | 使用场景 | 核心诉求 |
|---------|---------|---------|
| AI部门开发者 | 日常开发、AI能力调用 | 快速接入AI能力，减少重复开发 |
| IT部门运维 | 系统运维、安全管理 | 统一管控、安全审计、成本可控 |
| 产品经理 | 工作流配置、效果评估 | 低代码编排、数据驱动决策 |
| 上层应用(P1-P7) | 调用AI能力API | 稳定可靠的API服务、低延迟 |
| 管理层 | 成本与效果监控 | 投入产出可视化 |

### 1.4 核心业务目标

| 目标 | 量化指标 | 达标时间 |
|------|---------|---------|
| 全员工具覆盖 | 工具覆盖率100%、日活>=60% | Q1末 |
| 知识资产沉淀 | 知识库>=5000篇文档、5万+会话 | Q2末 |
| 能力平台化 | 6大模块全部上线 | Q3末 |
| 成本可控 | 模型调用成本降低>=30% | Q3末 |
| 安全合规 | 全链路审计覆盖、0起数据泄露 | Q4末 |
| 系统稳定 | 可用性>=99.9%、P99延迟<500ms | Q3末 |

### 2.2 六大模块详解

#### 模块一：大模型管理平台（M1）

**模块定位**：统一管理多个大语言模型的接入、评测、路由与成本控制。

**核心能力**：
- 多模型接入：支持DeepSeek、通义千问、GPT、Claude、WiNGPT等模型的统一接入
- 模型评测：针对特定业务场景（医疗问答、报告解读、话术生成等）进行准确率、响应时间、成本的对比评测
- 智能路由：根据任务类型、质量要求、成本预算自动选择最优模型
- 成本治理：实时监控每个模型、每个应用的调用量与费用，设置预算告警

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| 模型注册中心 | 模型元数据管理、API密钥管理、模型版本管理 |
| 模型评测引擎 | 自动化评测流水线、评测数据集管理、评测报告生成 |
| 智能路由引擎 | 路由策略配置、负载均衡、故障转移、A/B路由 |
| 成本治理中心 | 调用量统计、费用核算、预算管理、成本优化建议 |
| Prompt管理器 | Prompt模板管理、版本化、共享市场 |

---

#### 模块二：AI知识库平台（M2）

**模块定位**：企业知识与医疗知识的双栈管理平台，为RAG检索提供高质量知识底座。

**核心能力**：
- 双栈知识体系：企业知识库（产品手册、SOP、培训资料）+ 医疗知识库（临床指南、药品说明、诊疗规范）
- 智能文档处理：PDF/Word/Excel/图片等多格式文档的自动解析、切片、向量化
- 高精度检索：基于Milvus的向量检索 + BM25关键词检索的混合检索策略
- 知识图谱：医疗领域知识图谱构建，支持实体关系推理

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| 文档管理 | 文档上传、分类、标签、版本管理、权限控制 |
| 文档处理引擎 | 格式解析、智能切片（Chunk）、元数据提取 |
| 向量化引擎 | Embedding模型管理、批量向量化、增量更新 |
| 检索引擎 | 向量检索、关键词检索、混合检索、重排序（Reranker） |
| 知识图谱引擎 | 实体抽取、关系构建、图谱查询、图谱可视化。v1.0起引入Neo4j构建医疗知识图谱，v1.5实现GraphRAG三路融合检索（向量+关键词+图谱），详见优化方案11 |
| 知识质量管理 | 知识审核、过期检测、质量评分、更新提醒 |

---

#### 模块三：AI网关中台（M3）

**模块定位**：所有AI能力调用的统一入口，负责认证鉴权、流量治理、调用审计。

**核心能力**：
- 统一鉴权：基于API Key + JWT Token的双重认证，支持SSO集成
- 调用审计：全链路日志记录，包括请求/响应内容、模型选择、耗时、成本等
- 流量治理：限流、熔断、降级策略，防止单一应用过载
- 内容安全：输入/输出内容审核，敏感信息过滤

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| 认证鉴权 | API Key生命周期管理、JWT Token签发/验证、Keycloak SSO集成（OAuth2/OIDC）、MFA多因素认证、WebAuthn生物识别（P3/P4）。详见补充方案06 |
| 流量控制 | 请求限流（滑动窗口）、熔断降级（Sentinel）、请求排队 |
| 调用审计 | 请求日志记录、响应日志记录、异常追踪、合规审查 |
| 内容安全 | 输入内容过滤、输出内容审核、敏感信息脱敏、违规内容拦截 |
| 计费引擎 | 按调用计费、Token消耗统计、账单生成 |
| API版本管理 | URL路径版本化（/api/v1/）、多版本并行策略（最多N和N-1）、Breaking Change管控、SDK版本对应。详见优化方案10 |

---

#### 模块四：AI工作流平台（M4）

**模块定位**：可视化的AI工作流编排平台，支持复杂AI任务的流程化管理。

**核心能力**：
- 可视化编排：拖拽式工作流设计器，支持LLM调用、知识检索、条件判断、循环等节点
- 节点管理：预置节点库（LLM节点、RAG节点、HTTP节点、代码节点、判断节点等）
- 版本管理：工作流版本化，支持灰度发布和回滚
- 定时触发：支持Cron表达式的定时工作流触发

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| 工作流设计器 | 可视化拖拽编排、节点连接、参数配置 |
| 节点管理器 | 内置节点库、自定义节点开发、节点版本管理 |
| 执行引擎 | 工作流实例化、异步执行、错误重试、超时处理 |
| 版本管理 | 工作流版本化、发布/回滚、灰度策略（K8s Ingress Canary + Feature Flag，详见优化方案12） |
| 触发管理 | API触发、定时触发、事件触发、Webhook |
| 调试控制台 | 在线调试、断点调试、日志查看、性能分析 |

---

#### 模块五：智能体平台（M5）

**模块定位**：Agent智能体的编排与管理平台，支持单Agent和多Agent协作场景。

> **Agent框架选型决策**（详见补充方案02）：
> - Python层：LangChain 0.3+（基础编排+RAG）+ LangGraph 0.2+（Agent+工作流状态机）
> - Java层：Spring AI 1.0+（Spring原生集成）
> - AutoGen暂不引入，v2.5再评估

**核心能力**：
- Agent编排：定义Agent的角色、目标、可用工具、知识范围
- 多Agent协作：支持多个Agent的任务分解、协作与结果汇总
- 工具调用：Agent可调用API、数据库查询、文件操作等外部工具
- 记忆管理：短期记忆（会话上下文）+ 长期记忆（向量化存储）

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| Agent设计器 | Agent角色定义、工具配置、知识范围设置 |
| 工具注册中心 | API工具注册、参数定义、权限管理 |
| 协作编排器 | 多Agent任务分解策略、协作模式配置 |
| 记忆管理器 | 会话记忆、长期记忆、记忆检索、记忆清理 |
| Agent市场 | Agent模板库、共享与复用 |

---

#### 模块六：评测与监控平台（M6）

**模块定位**：AI系统的质量评测、效果监控与持续优化平台。

**核心能力**：
- 质量评测：自动化评测流水线，支持准确率、召回率、响应时间等多维度评测
- 效果监控：实时监控AI系统的关键指标，及时发现效果退化
- A/B测试：支持Prompt、模型、RAG策略等维度的A/B测试
- 告警通知：异常指标自动告警，支持企微/短信/邮件通知

**子模块划分**：

| 子模块 | 功能说明 |
|--------|---------|
| 评测引擎 | 评测数据集管理、评测任务编排、自动化执行 |
| 指标监控 | 实时指标面板、趋势分析、基线对比 |
| A/B测试框架 | 实验配置、流量分流、结果分析、统计显著性检验 |
| 告警中心 | 告警规则配置、告警通知（企微/短信/邮件）、告警历史 |
| 报告生成 | 日报/周报/月报自动生成、关键指标摘要 |
| 数据标注平台 | 标注任务管理、标注质量控制、标注数据导出 |

### 6.2 API规范

- **协议**：RESTful API，HTTPS强制
- **认证**：Header `Authorization: Bearer <API_KEY>`
- **版本**：URL路径版本化（`/api/v1/`）
- **响应格式**：JSON，统一包装结构

```json
{
  "code": 200,
  "message": "success",
  "data": { },
  "traceId": "xxx-xxx-xxx",
  "timestamp": 1710000000000
}
```

- **错误码**：遵循HTTP标准（400/401/403/404/429/500），业务错误码1000-9999
- **分页**：`?page=1&size=20`，返回`total`字段
- **流式响应**：Server-Sent Events (SSE)，`Content-Type: text/event-stream`

### 8.3 CI/CD流程

```
代码提交 → GitLab CI Pipeline
    │
    ├── 阶段1：代码检查
    │   ├── ESLint/Checkstyle 代码规范
    │   ├── SonarQube 代码质量
    │   └── 单元测试 (覆盖率>=80%)
    │
    ├── 阶段2：构建
    │   ├── Maven构建 (后端)
    │   ├── Vite构建 (前端)
    │   └── Docker镜像构建+推送
    │
    ├── 阶段3：部署测试环境
    │   ├── K8s Rolling Update
    │   └── 自动化集成测试
    │
    ├── 阶段4：部署预发布环境 (手动触发)
    │   ├── K8s Rolling Update
    │   └── 压测 + 回归测试
    │
    └── 阶段5：部署生产环境 (手动触发+审批)
        ├── 灰度发布 (10% → 30% → 100%)
        ├── 健康检查
        └── 自动回滚 (失败时)
```

### 8.4 灰度发布策略

| 阶段 | 流量比例 | 持续时间 | 观察指标 | 回滚条件 |
|------|---------|---------|---------|---------|
| 灰度1 | 10% | >=30分钟 | 错误率、延迟、业务指标 | 错误率>1%或P99>5s |
| 灰度2 | 30% | >=1小时 | 错误率、延迟、业务指标 | 错误率>0.5%或P99>3s |
| 全量 | 100% | - | 持续监控 | 任何异常立即回滚 |
