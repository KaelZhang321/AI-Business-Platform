## 二、灰度发布实施方案

### 2.1 技术选型

| 方案 | 实现方式 | 适用场景 | 复杂度 |
|------|---------|---------|--------|
| **K8s Ingress灰度（推荐）** | Ingress Annotation + Canary | 后端服务灰度 | 低 |
| Istio Service Mesh | VirtualService + DestinationRule | 细粒度流量治理 | 高 |
| Nginx Lua | 自定义Lua脚本 | 前端灰度 | 中 |
| Feature Flag | 代码级开关 | 功能级灰度 | 低 |

**推荐方案**：K8s Ingress灰度 + Feature Flag组合

### 2.2 K8s Ingress灰度配置

```yaml
# 灰度Ingress - 按用户ID Header路由
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: p8-gateway-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    # 方式1：按Header精确匹配（内测用户）
    nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"
    nginx.ingress.kubernetes.io/canary-by-header-value: "true"
    # 方式2：按权重百分比
    # nginx.ingress.kubernetes.io/canary-weight: "10"
spec:
  rules:
  - host: ai-api.lizkal.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: p8-gateway-canary  # 灰度版本Service
            port:
              number: 8080
```

### 2.3 灰度发布流程

```
┌──────────────────────────────────────────────┐
│           灰度发布标准流程                      │
│                                               │
│  ① 灰度准备                                   │
│     → 部署canary版本（新Deployment）           │
│     → 配置canary Ingress                       │
│     → 自动化冒烟测试通过                        │
│                                               │
│  ② 内测灰度（Header路由）                      │
│     → 仅Header X-Canary:true的请求到新版本     │
│     → 内部团队验证（3-5人，1天）                │
│     → 监控：错误率、延迟、业务指标               │
│                                               │
│  ③ 小流量灰度（10%权重）                       │
│     → canary-weight: 10                        │
│     → 持续≥30分钟                              │
│     → 告警条件：错误率>1% 或 P99>5s             │
│                                               │
│  ④ 中流量灰度（30%权重）                       │
│     → canary-weight: 30                        │
│     → 持续≥1小时                               │
│     → 告警条件：错误率>0.5% 或 P99>3s           │
│                                               │
│  ⑤ 全量发布（100%）                            │
│     → 更新主Deployment镜像                     │
│     → 删除canary Deployment和Ingress           │
│     → 持续监控24小时                            │
│                                               │
│  回滚：                                        │
│     → 任何阶段触发告警 → 立即 canary-weight:0   │
│     → 删除canary Deployment                    │
│     → 回滚耗时 < 1分钟                         │
└──────────────────────────────────────────────┘
```

### 2.4 Feature Flag服务

```java
// Feature Flag配置（Nacos动态配置）
@ConfigurationProperties(prefix = "feature")
public class FeatureFlags {
    private boolean graphRagEnabled = false;      // GraphRAG融合
    private boolean semanticCacheEnabled = false;  // 语义缓存
    private boolean newCopilotUI = false;          // 新版Copilot UI
    private boolean multiAgentEnabled = false;     // 多Agent协作

    // 按用户白名单控制
    private Set<String> betaUsers = Set.of();      // Beta测试用户ID
}

// 使用方式
@Service
public class AIService {
    @Autowired
    private FeatureFlags flags;

    public SearchResults search(String query, String userId) {
        if (flags.isGraphRagEnabled() || flags.getBetaUsers().contains(userId)) {
            return graphRagSearch(query);  // 新版GraphRAG
        }
        return ragSearch(query);  // 旧版纯RAG
    }
}
```

## 三、微前端评估

### 3.1 当前前端架构

```
当前：8个独立前端应用，各自独立构建部署

┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ P1 App │ │ P2 App │ │ P3 App │ │ P4 App │
│ React  │ │ React  │ │ React  │ │ React  │
└────────┘ └────────┘ └────────┘ └────────┘
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│ P5 App │ │P6 App  │ │ P7 App │ │ P8 App │
│ React  │ │ Vue 3  │ │ React  │ │ React  │
└────────┘ └────────┘ └────────┘ └────────┘

问题：
- 8次独立登录（SSO方案06已解决认证问题）
- 导航栏/通知中心/用户信息各自实现
- 切换产品需要跳转页面刷新
```

### 3.2 微前端方案对比

| 方案 | 技术实现 | 优势 | 劣势 | 适合度 |
|------|---------|------|------|--------|
| **qiankun** | 基于single-spa封装 | 成熟稳定，支持Vue+React | 样式隔离复杂，沙箱性能损耗 | ⭐⭐⭐ |
| **Module Federation** | Webpack 5原生 | 共享依赖高效，运行时加载 | 仅支持同构（都用Webpack） | ⭐⭐⭐⭐ |
| **iframe** | 原生隔离 | 天然沙箱，零冲突 | 通信复杂，性能差，SEO差 | ⭐⭐ |
| **不使用微前端** | 保持独立应用 | 零改造成本，简单 | 切换刷新，重复组件 | ⭐⭐⭐ |

### 3.3 评估结论：暂不引入微前端

**理由**：

| 考量 | 分析 |
|------|------|
| 用户切换频率 | 各岗位用户主要使用1-2个产品，跨产品切换不频繁 |
| 团队规模 | 5人团队，微前端增加的复杂度与人力不匹配 |
| 共享需求 | SSO已解决认证，共享导航可通过共享npm包实现 |
| Vue+React混用 | qiankun可支持，但增加调试复杂度 |
| 性能影响 | 微前端运行时开销对医疗场景不友好（P3要求≤1.5s） |

**替代方案**：

```
┌──────────────────────────────────────────────┐
│         轻量级前端整合方案                      │
│                                               │
│  共享Shell（不是微前端）：                      │
│                                               │
│  ┌──────────────────────────────────────┐     │
│  │ 共享npm包 @lizkal/shell               │     │
│  │ ├── TopNav组件（产品切换+通知+用户）   │     │
│  │ ├── SSOClient（统一认证）             │     │
│  │ ├── NotificationCenter（通知中心）    │     │
│  │ └── ThemeProvider（统一主题）          │     │
│  └──────────────────────────────────────┘     │
│                                               │
│  各产品引用Shell包：                            │
│  import { TopNav, SSOClient } from '@lizkal/shell'│
│                                               │
│  产品切换方式：                                 │
│  TopNav下拉菜单 → window.location跳转          │
│  （SSO Token不丢失，无需重新登录）               │
└──────────────────────────────────────────────┘
```

### 3.4 远期评估条件

当满足以下条件时，重新评估微前端引入：
