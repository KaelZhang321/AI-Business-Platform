# 优化方案13_AI 对话全链路性能与前端接入改造方案

> 版本：V2.0（基于架构评审重生成）  
> 日期：2026-04-29  
> 范围：`frontend/` + `ai-gateway/` 的 AI 对话、意图识别、API Query、前端 JSON 组件消费链路  
> 重要边界：**前端参与请求上下文、分段体验和渲染状态改造；UI JSON 组件仍由 AI 网关生成，生成位置不迁移到前端。**  
> 排除项：本方案不展开会议 BI 专项优化、不展开安全专项整改、不深入 Java 业务服务架构和数据库 schema 改造。  
> 依据文档：`docs/AI网关架构评审报告.md`（4 路并行静态分析）、`.omc/plans/p0p1-improvements-plan.md`（Ralplan 共识计划 v2）

> 设计约束（来自 Ralplan 共识 + 架构评审）：  
> 1. **代码是事实来源**，文档与实现冲突时以代码为准；  
> 2. **尽量保持现有公开请求/响应契约不变**，除非后续执行阶段明确批准改协议；  
> 3. **先加测试护栏，再做 `/chat` 重构**；  
> 4. `/chat` 的 **SSE 正确性与 traceId 稳定性** 是 P0，不应被更大范围的架构整理掩盖；  
> 5. **sync-in-async 阻塞** 是当前性能问题的系统性根因（评审报告 5.2 节），修复优先级最高；  
> 6. DI 统一需要明确 ChatWorkflow 的注入链，避免 BGE-M3 模型重复加载（最高风险变更）。

## 1. 背景与目标

当前 AI 对话从用户输入到前端组件 JSON 返回，整体链路偏长，且意图识别不够明确。结合当前代码（194 个 Python 文件、40+ API 端点）、4 路并行架构评审（覆盖核心对话流 / 业务核心 / 基础设施 / 外围模块）以及 Ralplan 共识计划，主要问题不是单一模型慢，而是链路上存在 **系统性的 sync-in-async 阻塞**、多次 LLM 调用、上下文不足、简单问题进入复杂编排、前端缺少分阶段消费能力等问题。

### 1.1 架构评审核心数据

| 维度 | 当前评分 | 主要问题 |
|------|---------|---------|
| **性能** | **4/10** | 5 处 CRITICAL sync-in-async 阻塞事件循环；意图分类无缓存/快路径 |
| 可维护性 | 6/10 | 3 处 DI 绕过；config 199 行单类膨胀；_normalize_text 5 处重复 |
| 安全性 | 5/10 | 12 个敏感字段无 SecretStr；CORS 全开放；MCP 无鉴权 |
| 可测试性 | 6/10 | MCP tools 零测试；chat.py 模块级硬实例化难替换 |
| 可观测性 | 6/10 | 无 OpenTelemetry；Prometheus 高基数标签 |
| 可扩展性 | 6/10 | AppResources 管理 18 实例趋向 God Object |

本方案目标：

1. 降低 AI 对话与 `/api-query` 的端到端耗时（性能 4→7 分）；
2. 提升 Python 侧意图识别稳定性；
3. 增加前端上下文透传、加载状态、分段响应消费能力；
4. 保持 UI JSON 生成职责仍在 `ai-gateway`，前端只消费和渲染；
5. 为后续实施提供阶段、文件范围、验收标准和风险控制。

## 3. 核心问题诊断（基于架构评审精确发现）

### 3.1 系统性 sync-in-async 阻塞（性能根因）

架构评审发现 **5 处 CRITICAL 级同步调用** 直接在 async 函数中执行，阻塞 asyncio 事件循环：

| 位置 | 同步调用 | 阻塞时长 | 影响 |
|------|---------|---------|------|
| `rag_service.py:124` | `embedder.encode([query])` | 50-200ms | 所有 KNOWLEDGE 请求 |
| `rag_service.py:126` | `collection.search()` | 100-500ms | 所有 KNOWLEDGE 请求 |
| `rag_service.py:279` | `reranker.compute_score(pairs)` | 100-500ms | 所有 KNOWLEDGE 请求 |
| `semantic_cache.py:115` | `embedder.encode([text])` | 50-200ms | 语义缓存开启时 |
| `semantic_cache.py:138` | `collection.search()` | 100-500ms | 语义缓存开启时 |

**关键发现**：三路"并行"检索（`rag_service.py:97-106` 的 `asyncio.gather`）中 `_vector_search` 内部是同步的，**实际三路是串行执行**（只有 ES 和 Neo4j 是真正异步）。

**正面参考**：`generic_query_executor.py:90` 正确使用了 `asyncio.to_thread(vn.ask, question)`。

### 3.2 意图识别过度依赖 LLM

当前 `IntentClassifier.classify()`（`intent_classifier.py:73`）：

- 每次请求 **必须** 先调 LLM（`intent_classifier.py:105-111`，延迟 1-30s）
- 关键词兜底 **仅在 LLM 失败时** 触发（`intent_classifier.py:87-89`），而非作为优先的低延迟路径
- 无意图缓存（相同/相似问题重复分类）
- `/api-query` 的 `ApiParamExtractor.route_query()` 也以 LLM 为主要判断手段

### 3.3 DI 绕过导致资源重复创建

架构评审发现 **3 处路由层绕过 AppResources 依赖注入**：

| 位置 | 问题 | 后果 |
|------|------|------|
| `chat.py:12` | `workflow = ChatWorkflow()` 模块级硬实例化 | BGE-M3 可能加载 3 份（RAG + SemanticCache + AppResources 各一份），浪费 2-6GB 内存 |
| `smart_meal.py:26-27` | 模块级 `SmartMealRiskService()` / `SmartMealPackageRecommendService()` | 额外创建 4 个 MySQL 连接池（2 ODS + 2 Business），shutdown 时不被关闭 |
| `api_query.py:64-73` | 7+ 进程级单例 | 与 FastAPI lifespan 事件脱耦 |

### 3.4 连接池碎片化

当前系统存在 **9+ 个独立 MySQL 连接池**，其中 5 个不在 AppResources 生命周期管理中：

| 连接池 | 最大连接数 | 生命周期管理 |
|--------|-----------|------------|
| AppResources business_pool | 配置化 | ✓ AppResources |
| Meeting BI sync session | 30 | ✓ SQLAlchemy |
| Meeting BI async pool | 5 | ✓ aiomysql |
| Vanna 内部连接 | 不受控 | ⚠ 无管理 |
| HealthQuadrant ODS pool | 配置化 | ✓ AppResources |
| SmartMeal ODS pool ×2 | 3 each | ⚠ 无管理 |
| SmartMeal Business pool ×2 | 3 each | ⚠ 无管理 |
| MCP Text2SQL pool | 独立创建 | ⚠ 无管理 |

### 3.5 简单查询进入复杂链路

`/api-query` 架构成熟（DAG 引擎具备步骤级+图级双层超时保护），但链路较重：

```text
route_query → retrieve_candidates → build_plan → validate_plan → execute_plan → build_response
```

对于单接口、高置信、固定模板场景，不应进入完整 Planner 链路。

### 3.6 UI JSON 返回等待完整结果

当前前端多数调用等待完整响应后再展示结果。即便后端部分阶段已完成，前端也无法先展示意图、候选、数据或骨架组件。

### 3.7 前端上下文透传不一致

`AIReportComparisonReportView` 已传递 `reportContext`、`availableMetrics`、`targetIds` 等上下文，但 `ConsultantAIWorkbench` 当前只传 `{ query }`。

### 3.8 `/chat` 流式正确性存在 P0 问题

- `ChatWorkflow.stream()`（`chat_workflow.py:106-110`）基于 LangGraph `astream_events`，但只在 `on_node_end` 时发送完整节点结果（`chat_workflow.py:127-133`）——**非真正 token 级流式**
- `ModelRouter.stream_chat()`（`model_router.py:149`）已实现流式，但 ChatWorkflow 的 chat 节点未使用
- `_convert_event()` generator 存在嵌套 yield 风险
- route 层和 workflow 可能都发 `STREAM_END`，导致重复终止事件
- 首轮 `traceId` 可能与最终 `conversation_id` 不一致

### 3.9 `/chat` 与 `/api-query` 的组合方式尚未统一

当前至少存在三种组合方式：

1. `chat.py` 模块级单例 `ChatWorkflow()`
2. `query.py` 通过 `AppResources` / route dependencies 注入共享服务
3. `/api-query` 使用进程级 workflow/service singleton

### 3.10 基础设施层问题

- **lifespan 全串行初始化**（`main.py:165-235`）：Milvus/ES 验证串行 + 4 个 warmup 串行，可用 `asyncio.gather` 节省 2-5s
- **配置安全**：12 个敏感字段（含 `rabbitmq_url`）均为普通 `str`（`config.py`），未用 `SecretStr`；CORS `allow_origins=["*"]`（`main.py:346`）
- **Prometheus 高基数标签**（`main.py:467`）：使用原始 URL 路径而非路由模板
- **MCP Server 循环导入风险**：`tools.py:29` 反向 import `app.main.app`
- **BGE-M3 懒加载无锁**：`rag_service.py:53-68` 的 `_embedder()` 无 `threading.Lock`，并发首请求可能加载多份模型

## 4. 目标架构

### 4.1 总体链路

```text
前端 AI 输入
  → 统一 AI 请求服务封装
  → 携带 pageContext / entityContext / conversationContext
  → ai-gateway FastIntentRouter
      ├─ 规则高置信命中：跳过 LLM route（<1ms）
      ├─ 意图缓存命中：跳过 LLM route（<1ms）
      ├─ API Catalog 快速候选命中：进入 fast path
      └─ 低置信/复杂问题：进入 LLM route 或澄清
  → ApiQueryWorkflow
      ├─ fast path：单接口 + 参数抽取 + 模板 UI
      └─ planner path：多接口 DAG + 执行图 + 响应构建
  → DynamicUIService / ResponseBuilder 生成 UI JSON
  → 前端按阶段消费并渲染
```

### 4.2 性能目标（基于延迟风险分级）

| 步骤 | 当前风险 | 目标 |
|------|---------|------|
| 意图分类 | CRITICAL (1-30s LLM) | 高频明确问题 <10ms（规则优先+缓存） |
| RAG 三路检索 | CRITICAL (sync串行) | 真正并行（asyncio.to_thread），总耗时 ≈ 最慢单路 |
| BGE-Reranker | CRITICAL (sync阻塞) | to_thread 包装，不阻塞事件循环 |
| 语义缓存 | HIGH (sync阻塞) | to_thread 包装 |
| UI JSON 构建 | LOW | 模板优先 <100ms |

### 4.3 职责边界

| 模块 | 改造后职责 |
|---|---|
| 前端 | 采集上下文、发起请求、展示阶段状态、消费后端 UI JSON、处理澄清选项 |
| `ChatWorkflow` | 通用对话入口与业务阶段事件编排，通过 provider 注入依赖 |
| `ApiQueryWorkflow` | 自然语言业务查询主编排，支持 fast path / planner path |
| `FastIntentRouter` | Python 侧规则、缓存、上下文和候选辅助识别 |
| `DynamicUIService` | 继续负责 UI JSON 生成、模板优先和 Guard 校验 |
| 前端渲染层 | 只消费 `ui_spec`，不生成 `ui_spec` |

### 4.4 `/chat` 组合方式决策表

| 方案 | 形态 | 优点 | 缺点 | 建议 |
|---|---|---|---|---|
| Route dependency provider + AppResources | `Depends(get_chat_workflow)` | 与 `query.py` 风格一致 | 可能把 AppResources 变成 service locator | 可评估 |
| **Dedicated workflow factory/provider** | AppResources 只管共享资源，workflow 由独立 provider 构造 | **资源归属清晰**、组装可测试 | 新增一层组合逻辑 | **推荐** |
| 保留模块级 singleton | `workflow = ChatWorkflow()` | 改动最小 | 生命周期/测试替换/资源关闭脆弱 | 仅最小改动时 |

**关键约束**（Architect 审查补充）：无论选择哪种方案，ChatWorkflow 构造函数必须接受 `rag_service`, `intent_classifier`, `llm_service`, `semantic_cache` 等注入参数，使其复用 AppResources 中的共享实例。这是解决 BGE-M3 模型 3 份加载问题的唯一途径。

## 5. 后端改造方案

### 5.0 先加测试护栏，再做结构重构

在改动 `/chat` 流式逻辑、DI 组合方式、会话语义前，必须先补测试：

```text
ai-gateway/tests/api/test_chat_routes.py
ai-gateway/tests/services/test_chat_workflow.py
```

第一阶段测试目标：

1. 锁定非流式 `/api/v1/chat` 成功与失败边界；
2. 锁定当前 `STREAM_*` SSE 包结构（version/id/traceId/timestamp/source/type/payload）；
3. 把已知 P0 流式缺陷标记为严格 `xfail`：nested-generator flattening / duplicate terminal `STREAM_END`。

### 5.1 修复 RAG 同步阻塞（P0）

将以下同步调用包装为 `asyncio.to_thread()`：

| 文件:行号 | 同步调用 | 修复方式 |
|----------|---------|---------|
| `rag_service.py:124` | `embedder.encode([query])` | `await asyncio.to_thread(embedder.encode, [query])` |
| `rag_service.py:126` | `collection.search(...)` | `await asyncio.to_thread(collection.search, ...)` |
| `rag_service.py:279` | `reranker.compute_score(pairs)` | `await asyncio.to_thread(reranker.compute_score, pairs)` |
| `semantic_cache.py:115` | `embedder.encode([text])` | `await asyncio.to_thread(embedder.encode, [text])` |
| `semantic_cache.py:138` | `collection.search(...)` | `await asyncio.to_thread(collection.search, ...)` |

**同时修复懒加载竞争**（Ralplan Pre-Mortem #4）：

```python
# rag_service.py — _embedder() 和 _reranker() 增加锁保护
import threading
_model_lock = threading.Lock()

@property
def _embedder(self):
    if self.__embedder is None:
        with _model_lock:
            if self.__embedder is None:  # double-check
                self.__embedder = FlagModel(...)
    return self.__embedder
```

### 5.2 增加 `FastIntentRouter`（P0）

新增服务：`ai-gateway/app/services/api_query_fast_intent_router.py`

识别优先级：
1. 页面上下文强匹配
2. 业务关键词和 synonym 规则
3. 进程内 TTL 意图缓存（基于 message hash）
4. API Catalog topK 反向推断
5. LLM 兜底

同时改造 `IntentClassifier.classify()`：
- 先执行关键词匹配（<1ms），高置信度直接返回（快路径）
- 关键词低置信度时才调 LLM
- LLM 结果写入 LRU 缓存（TTL 可配置）

### 5.3 增加阶段耗时观测

新增统一计时工具：`ai-gateway/app/utils/timing.py`

核心指标：

```text
intent_classify_ms          # 意图分类耗时
fast_intent_router_ms       # 快速路由耗时
rag_embedding_ms            # Embedding 编码耗时
rag_vector_search_ms        # 向量检索耗时
rag_keyword_search_ms       # 关键词检索耗时
rag_graph_search_ms         # 图谱检索耗时
rag_rerank_ms               # 重排序耗时
api_query_route_ms          # API 查询路由耗时
catalog_retrieve_ms         # 候选检索耗时
param_extract_ms            # 参数抽取耗时
planner_ms                  # DAG 规划耗时
execution_ms                # 执行耗时
ui_spec_build_ms            # UI Spec 构建耗时
llm_call_count              # LLM 调用次数
chat_stream_first_event_ms  # SSE 首事件耗时
chat_stream_end_ms          # SSE 完成耗时
```

### 5.4 `/api-query` 快慢双通道

#### Fast Path

适用：fast intent 高置信命中 + 候选接口唯一 + 无复杂 predecessor + 可使用固定模板。

```text
prepare_request → fast_route → retrieve_candidates → extract_params
  → build_single_step_plan → execute_plan → build_template_response
```

#### Planner Path

适用：多候选/多接口依赖/组合结果/低置信但可由 LLM 仲裁。流程保持现有。

### 5.5 低置信度澄清机制

新增响应状态 `WAIT_CLARIFY`：

```json
{
  "execution_status": "WAIT_CLARIFY",
  "message": "您是想查询客户档案，还是查看体检报告？",
  "options": [
    { "label": "客户档案", "query_domains": ["customer_profile"] },
    { "label": "体检报告", "query_domains": ["patient_exam"] }
  ]
}
```

约束：低置信澄清优先加在 `/api-query` 路径；`/chat` 是否直接透出澄清选项属后续协议决定。

### 5.6 UI JSON 模板优先

```text
UI JSON 仍由 ai-gateway 的 DynamicUIService / ResponseBuilder 生成。
前端不迁移 UI JSON 生成职责。
```

处理顺序：已知业务域+已知数据形状 → 固定模板 → 规则生成 → LLM UI 仅兜底。

### 5.7 `/chat` SSE 正确性修复（P0）

目标：

1. `ChatWorkflow.stream()` 输出可直接消费的事件，不是嵌套 generator
2. 成功流只发一次 `STREAM_END`
3. 异常流统一发一次 `STREAM_ERROR`
4. 单次请求 `traceId` 全流内保持稳定
5. 首轮是否暴露生成后的 `conversation_id`，不在 P0 中强行改协议

涉及文件：`chat_workflow.py`、`chat.py`、`test_chat_routes.py`、`test_chat_workflow.py`

## 6. 前端实现改造方案

### 6.1 新增统一 AI Query 客户端

### 6.2 改造 `ConsultantAIWorkbench`

从 `apiClient.post('/api/v1/api-query', { query })` 改为：

```ts
aiQueryApi.query({
  query,
  context: {
    page: 'consultant_workbench',
    module: 'customer_profile',

    visibleCards: dashboardCards.map(card => card.id),
```

改造点：增加 `conversationId` 状态、透传 `selectedCustomer`、显式请求状态（idle|routing|loading|rendering|error）、支持 `WAIT_CLARIFY` 展示。

### 6.3 改造主助手

将 `App.tsx` 的 `handleSendMessage()` 从 `setTimeout` mock 改为真实请求。`AIAssistant` 增加 loading/error/clarify 展示。

### 6.4 增加分阶段响应消费

前端状态机：

```ts
type AiRequestStage = 'idle' | 'routing' | 'retrieving' | 'executing' | 'rendering' | 'clarifying' | 'done' | 'error';
```

`/chat` SSE 消费先按"阶段流式"语义实现：

```text
STREAM_START  → 进入处理中
STREAM_CHUNK  → 追加阶段文本/sources/ui_spec
STREAM_END    → 收尾
STREAM_ERROR  → 展示错误
```

## 11. 最终结论

本方案的核心不是把 UI JSON 生成挪到前端，而是：

1. **消除系统性 sync-in-async 阻塞**（性能根因，5 处 CRITICAL 级修复）
2. **让意图识别走快路径**（关键词优先 + 缓存，高频问题 <10ms）
3. **统一 DI 体系**（消除 BGE-M3 模型 3 份加载，节省 2-6GB 内存）
4. **修正 `/chat` 流式正确性**（单次 STREAM_END、稳定 traceId）
5. **前端标准化上下文透传**（page/module/entity/conversation）

最终目标链路：

```text
前端标准化上下文
  → Python 快速意图识别（规则优先，<10ms）
  → 简单查询 fast path（跳过 Planner）
  → 复杂查询 planner path（现有 DAG 引擎）
  → `/chat` 阶段流式正确输出
  → AI 网关模板优先生成 UI JSON
  → 前端分阶段消费和渲染
```

这样既能将性能评分从 4/10 提升至 7/10，又能先把 `/chat` 的流式正确性和组合边界修稳，同时保持现有动态 UI 架构边界不被打乱。
