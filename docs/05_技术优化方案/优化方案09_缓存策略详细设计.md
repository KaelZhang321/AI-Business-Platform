## 一、缓存分层架构

### 1.1 三级缓存体系

```
┌──────────────────────────────────────────────────┐
│                  缓存分层架构                      │
│                                                   │
│  L1 本地缓存（JVM进程内）                          │
│  ├── Caffeine Cache                               │
│  ├── 容量：每节点100MB                             │
│  ├── 场景：热点配置、字典数据、模型路由表            │
│  └── TTL：5-30分钟                                 │
│                                                   │
│  L2 分布式缓存（Redis集群）                        │
│  ├── Redis 7.x 哨兵模式（3节点）                   │
│  ├── 容量：8GB                                     │
│  ├── 场景：会话上下文、客户画像、API结果缓存         │
│  └── TTL：5分钟-1周                                │
│                                                   │
│  L3 CDN/浏览器缓存                                 │
│  ├── 阿里云CDN + Service Worker                    │
│  ├── 场景：静态资源、AI科普内容、图片                │
│  └── TTL：1天-30天                                 │
│                                                   │
│  查找顺序：L1 → L2 → DB/API → 回写L2+L1          │
└──────────────────────────────────────────────────┘
```

## 二、缓存键设计规范

### 2.1 键命名规则

```
{产品}:{模块}:{实体}:{ID}:{版本/维度}

示例：
p8:rag:result:{query_hash}          # RAG检索结果缓存
p8:model:response:{request_hash}    # 模型调用结果缓存
p2:patient:profile:{patient_id}     # 患者画像缓存
p3:nurse:schedule:{date}:{dept_id}  # 排班缓存
p6:customer:portrait:{external_id}  # 企微客户画像
p8:config:model_routing             # 模型路由配置
```

### 2.2 Hash计算策略

```java
// RAG/模型调用缓存键 - 语义相似请求命中
public class CacheKeyGenerator {

    // 精确匹配：完全相同的请求
    public static String exactKey(String product, String module, Object request) {
        String content = JSON.toJSONString(request);
        String hash = DigestUtils.md5DigestAsHex(content.getBytes());
        return String.format("%s:%s:exact:%s", product, module, hash);
    }

    // 模糊匹配：RAG检索的语义缓存（Embedding相似度>0.95命中）
    // 注意：语义缓存通过Milvus实现，不走Redis
    public static String ragCacheKey(String knowledgeBaseId, String queryHash) {
        return String.format("p8:rag:result:%s:%s", knowledgeBaseId, queryHash);
    }

    // 模型调用缓存：相同prompt+相同参数
    public static String modelCacheKey(String modelId, String promptHash,
                                        double temperature) {
        // temperature>0时不缓存（随机输出）
        if (temperature > 0) return null;
        return String.format("p8:model:response:%s:%s", modelId, promptHash);
    }
}
```

### 3.2 知识库更新→缓存同步机制

```
┌──────────────────────────────────────────────┐
│         知识库更新缓存同步流程                  │
│                                               │
│  文档上传/更新                                 │
│       │                                       │
│       ▼                                       │
│  文档处理Pipeline                              │
│  (解析→分片→Embedding→入库Milvus)              │
│       │                                       │
│       ▼                                       │
│  发布缓存失效事件（RabbitMQ）                   │
│  {                                            │
│    "event": "KB_UPDATED",                     │
│    "kb_id": "kb_001",                         │
│    "version": 42,                             │
│    "affected_docs": ["doc_1", "doc_2"],       │
│    "timestamp": 1710000000                    │
│  }                                            │
│       │                                       │
│       ├── Redis: DEL p8:rag:result:kb_001:*   │
│       │   （删除该知识库所有RAG缓存）            │
│       │                                       │
│       ├── Caffeine: 广播失效                   │
│       │   （通过Redis Pub/Sub通知所有节点）      │
│       │                                       │
│       └── 版本号更新                            │
│           SET p8:kb:version:kb_001 → 42       │
└──────────────────────────────────────────────┘
```

```java
// 缓存失效监听器
@RabbitListener(queues = "cache.invalidation")
public void handleCacheInvalidation(CacheInvalidationEvent event) {
    switch (event.getType()) {
        case "KB_UPDATED":
            // 清除该知识库的RAG缓存
            String pattern = "p8:rag:result:" + event.getKbId() + ":*";
            redisTemplate.delete(redisTemplate.keys(pattern));
            // 通知本地缓存失效
            localCacheManager.invalidate("rag", event.getKbId());
            break;
        case "PATIENT_UPDATED":
            redisTemplate.delete("p2:patient:profile:" + event.getEntityId());
            break;
        case "CONFIG_CHANGED":
            localCacheManager.invalidateAll("config");
            break;
    }
}
```

## 四、缓存防护方案

### 4.1 三大经典问题防护

```
┌──────────────────────────────────────────────┐
│          缓存防护架构                          │
│                                               │
│  ① 缓存穿透（查询不存在的数据）               │
│     ┌────────────────────────────────────┐    │
│     │ 防护：布隆过滤器 + 空值缓存           │    │
│     │                                     │    │
│     │ 请求 → 布隆过滤器判断               │    │
│     │   ├── 不存在 → 直接返回null          │    │
│     │   └── 可能存在 → 查Redis            │    │
│     │         ├── 命中 → 返回              │    │
│     │         └── 未命中 → 查DB            │    │
│     │               ├── 有数据 → 回写缓存  │    │
│     │               └── 无数据 → 缓存null  │    │
│     │                   (TTL=60s，防穿透)  │    │
│     └────────────────────────────────────┘    │
│                                               │
│  ② 缓存击穿（热点Key过期瞬间高并发）          │
│     ┌────────────────────────────────────┐    │
│     │ 防护：互斥锁（Redisson分布式锁）     │    │
│     │                                     │    │
│     │ Key过期 → 多请求并发到达             │    │
│     │   → 第一个请求获取锁 → 查DB → 回写  │    │
│     │   → 其余请求等待锁释放 → 读新缓存    │    │
│     │   锁超时：500ms（防死锁）            │    │
│     └────────────────────────────────────┘    │
│                                               │
│  ③ 缓存雪崩（大量Key同时过期）                │
│     ┌────────────────────────────────────┐    │
│     │ 防护：TTL随机化 + 预热 + 熔断        │    │
│     │                                     │
│     │ - TTL = baseTTL + random(0, 300s)   │    │
│     │ - 系统启动时预热热点数据              │    │
│     │ - 数据库QPS超阈值 → Sentinel熔断     │    │
│     │ - 降级：返回旧缓存/默认值             │    │
│     └────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

### 4.2 实现代码

```java
@Service
public class CacheProtectedService {

    @Autowired
    private RedissonClient redisson;

    @Autowired
    private RedisTemplate<String, Object> redis;

    @Autowired
    private BloomFilter<String> bloomFilter;

    /**
     * 带穿透+击穿防护的缓存查询
     */
    public <T> T getWithProtection(String key, Class<T> type,
                                    Supplier<T> dbFallback,
                                    long ttlSeconds) {
        // 1. 布隆过滤器防穿透
        if (!bloomFilter.mightContain(key)) {
            return null;
        }

        // 2. 查缓存
        T cached = (T) redis.opsForValue().get(key);
        if (cached != null) {
            return cached instanceof NullValue ? null : cached;
        }

        // 3. 互斥锁防击穿
        RLock lock = redisson.getLock("lock:" + key);
        try {
            if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
                try {
                    // 双重检查
                    cached = (T) redis.opsForValue().get(key);
                    if (cached != null) return cached;

                    // 4. 查数据库
                    T result = dbFallback.get();
                    if (result != null) {
                        // TTL随机化防雪崩
                        long randomTTL = ttlSeconds + ThreadLocalRandom.current()
                            .nextLong(0, 300);
                        redis.opsForValue().set(key, result, randomTTL,
                            TimeUnit.SECONDS);
                    } else {
                        // 缓存null值防穿透（短TTL）
                        redis.opsForValue().set(key, NullValue.INSTANCE, 60,
                            TimeUnit.SECONDS);
                    }
                    return result;
                } finally {
                    lock.unlock();
                }
            }
            // 未获取锁，等待后重试
            Thread.sleep(50);
            return (T) redis.opsForValue().get(key);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return dbFallback.get();
        }
    }
}
```

## 五、AI调用缓存策略（成本优化核心）

### 5.1 模型调用缓存规则

| 场景 | 是否缓存 | TTL | 缓存条件 |
|------|---------|-----|---------|
| RAG知识库检索 | ✅ | 1小时 | 同一知识库+相同query |
| 模型调用（temperature=0） | ✅ | 1小时 | 相同prompt+相同参数 |
| 模型调用（temperature>0） | ❌ | — | 输出不确定，不缓存 |
| 意图分类 | ✅ | 24小时 | 相同输入文本 |
| Embedding向量化 | ✅ | 7天 | 相同文本 |
| 流式响应 | ❌ | — | SSE流不适合缓存 |

### 5.2 语义缓存（Semantic Cache）

```
┌──────────────────────────────────────────────┐
│           语义缓存架构                         │
│                                               │
│  用户问题: "高血压吃什么药？"                   │
│       │                                       │
│       ▼                                       │
│  Embedding向量化                               │
│       │                                       │
│       ▼                                       │
│  Milvus检索已缓存的问题向量                     │
│  （相似度 > 0.95 → 命中）                       │
│       │                                       │
│       ├── 命中 → 直接返回缓存的回答              │
│       │   "高血压常见降压药有…"                  │
│       │                                       │
│       └── 未命中 → 正常RAG+模型调用             │
│           → 结果存入语义缓存                     │
│           {问题Embedding, 回答, TTL, KB版本}     │
└──────────────────────────────────────────────┘
```

**成本节省估算**：

| 指标 | 无缓存 | 精确缓存 | 精确+语义缓存 |
|------|--------|---------|-------------|
| 日均模型调用 | 10,000次 | 8,000次（-20%） | 6,000次（-40%） |
| 日均Embedding调用 | 10,000次 | 10,000次 | 7,000次（-30%） |
| 月API成本 | ¥15,000 | ¥12,000 | ¥9,000 |
| **月节省** | — | ¥3,000 | **¥6,000** |
