# 补充方案06：统一认证中心与零容错兜底架构

> 解决问题：（1）8产品缺少跨产品SSO方案；（2）配伍禁忌/剂量计算100%准确率缺少兜底机制
> 优先级：P1（合计3-4天工作量）

# Part A：统一认证中心（SSO）方案

## 一、需求分析

### 1.1 各产品认证需求

| 产品 | 认证方式 | 角色数 | 特殊要求 |
|------|---------|--------|---------|
| P1 | 账号密码 | 3 | — |
| P2 | 账号密码 | 3 | 等保三级 |
| P3 | 密码+**生物识别** | 3 | 双因子认证 |
| P4 | 密码+**生物识别** | 3 | 双因子认证 |
| P5 | 账号密码 | 3 | — |
| P6 | **企微免登** | 3 | 企微OAuth2.0 |
| P7 | 账号密码 | 3 | 数据源权限 |
| P8 | API Key + 账号 | 3 | API Key管理 |

### 1.2 核心挑战

1. **8个产品跨域SSO**：用户登录一次，全产品免登
2. **P3/P4双因子认证**：密码+指纹/面部识别
3. **P6企微免登**：企微环境内自动登录
4. **P8 API Key**：应用级认证 vs 用户级认证并存
5. **角色权限映射**：同一用户在不同产品中可能是不同角色

### 2.2 选型：Keycloak 24.x

| 维度 | Keycloak | CAS | 自研 |
|------|----------|-----|------|
| 协议支持 | OAuth2/OIDC/SAML | CAS/OAuth2 | 按需 |
| MFA/2FA | 内置WebAuthn/TOTP | 需扩展 | 需开发 |
| Identity Broker | 内置（企微/LDAP） | 需扩展 | 需开发 |
| 管理后台 | 完善 | 基础 | 需开发 |
| 学习曲线 | 中 | 低 | 高 |
| 社区 | GitHub 24K+ stars | 成熟 | — |
| **推荐** | **✅** | ⚠️ | ❌ |

### 2.3 认证流程

#### P1-P5/P7/P8（标准OIDC流程）

```
用户访问P1 → 检测无Token → 跳转Keycloak登录页
  → 输入账号密码 → (P3/P4额外WebAuthn验证)
  → Keycloak签发JWT（Access Token + Refresh Token）
  → 跳转回P1（携带code → 换取token）
  → 用户访问P2 → 检测到Keycloak SSO Session → 自动签发P2 Token
  → 无需再次登录
```

#### P6（企微免登流程）

```
用户打开P6侧边栏 → 企微JS-SDK获取临时code
  → P6 BFF携带code调用企微API获取userId
  → BFF携带userId调用Keycloak Identity Broker
  → Keycloak查找映射用户 → 签发JWT
  → P6前端使用JWT调用API
```

#### P8 API Key（应用级认证）

```
外部应用携带API Key请求P8 Gateway
  → Gateway验证API Key有效性+权限范围
  → 生成内部JWT（标记为service-account）
  → 转发请求至后端服务
```

# Part B：100%零容错兜底架构（药物安全三级防护）

## 一、问题定义

以下两项AI精度要求为**100%零容错**：

| 指标 | 目标 | 影响 | 后果 |
|------|------|------|------|
| 严重配伍禁忌检出率 | **=100%** | P2/P3 | 漏检→患者严重不良反应 |
| 药物剂量计算准确率 | **=100%** | P3 | 错误→过量/不足用药 |

**核心原则**：AI模型无法保证100%准确率，必须有独立于AI的兜底机制。

---

┌────────────────────────────────────────────────────────────┐

## 三、第一级：规则引擎详细设计

### 3.1 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 服务端规则引擎 | **Drools 8.x**（Java）| 高性能规则执行、支持热更新、Spring集成 |
| 前端本地规则 | **json-rules-engine**（JS）| P3离线模式使用，纯JS无依赖 |

### 3.2 规则数据结构

```json
{
  "ruleId": "CI-001",
  "ruleType": "CONTRAINDICATION",
  "severity": "CRITICAL",
  "action": "BLOCK",
  "drugA": { "atcCode": "J01D*", "name": "头孢类抗生素" },
  "drugB": { "atcCode": "V03AB*", "name": "含酒精制剂" },
  "mechanism": "双硫仑样反应（disulfiram-like reaction）",
  "consequence": "面部潮红、恶心呕吐、心动过速，严重可致休克",
  "evidence": "药品说明书+《中国药典》",
  "lastUpdated": "2026-01-15",
  "offlineAvailable": true
}
```

### 3.3 规则库规模

| 规则类型 | 数量 | 严重级别 | 处理方式 |
|---------|------|---------|---------|
| 严重配伍禁忌 | ~200条 | CRITICAL | **强制阻断** |
| 高危配伍警告 | ~500条 | HIGH | 弹窗警告+确认 |
| 一般交互提醒 | ~2000条 | MEDIUM | 信息提示 |
| 剂量上限规则 | ~500条 | CRITICAL | 超限阻断 |
| 特殊人群规则 | ~300条 | HIGH | 老人/儿童/孕妇/肾功能 |
| **合计** | **~3500条** | | |

### 3.4 规则更新机制

```
药品知识库更新 → 规则编辑（药剂科+AI部门）
  → 规则审核（药剂科主任签字）
  → 规则测试（自动化回归测试）
  → 规则发布（Drools热更新，无需重启）
  → 前端规则同步（P3每日凌晨拉取最新版本）
```

### 3.5 检查流程（处方提交时）

```
医生提交处方
  ↓
[第一级] 规则引擎检查（<100ms，同步阻断）
  ├── CRITICAL命中 → 弹窗"严重禁忌，不可提交" → 强制返回修改
  ├── HIGH命中 → 弹窗"高危警告" → 医生填写理由后可继续
  └── 通过 → 进入第二级
  ↓
[第二级] AI模型检查（<3s，异步）
  ├── AI发现风险 → 弹窗警告 → 医生确认或修改
  └── AI未发现 → 进入第三级
  ↓
[第三级] 药师审方（人工流程）
  ├── 药师发现问题 → 退回医生
  └── 药师通过 → 医生最终签名确认 → 处方生效
```

## 四、降级策略

| 场景 | 第一级规则引擎 | 第二级AI模型 | 第三级人工 |
|------|--------------|------------|-----------|
| 全部正常 | ✅ | ✅ | ✅ |
| AI服务不可用 | ✅ | ❌（跳过） | ✅ |
| 网络中断（P3离线） | ✅（本地规则） | ❌ | ❌（延迟审核） |
| 规则引擎异常 | ❌ | ✅ | ✅ + 人工全量审核 |
| 全部不可用 | — | — | **强制人工审方，不允许自动通过** |

**铁律**：任何情况下，处方不能绕过所有三级检查直接生效。至少有一级防护可用时才允许处方进入流程。
