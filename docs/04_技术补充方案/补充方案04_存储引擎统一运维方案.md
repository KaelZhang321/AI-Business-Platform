# 补充方案04：存储引擎统一运维方案

> 解决问题：9种存储引擎运维复杂度高，缺少统一管理方案
> 优先级：P2（2天工作量）
> 影响范围：全产品线数据安全与稳定性

---

## 一、存储引擎全景

### 1.1 当前8种存储引擎

| # | 引擎 | 版本 | 使用产品 | 数据类型 | 部署模式 |
|---|------|------|---------|---------|---------|
| 1 | MySQL | 8.0 | 全部业务线 | 业务数据（ACID） | 主从2节点 |
| 2 | Oracle | 19c | P7（只读） | 财务数据 | 已有（外部） |
| 3 | MongoDB | — | P5 | AI交互日志 | 单节点 |
| 4 | Milvus | 2.3.x | P1-P4/P8 | 向量数据 | 集群3节点 |
| 5 | Redis | 7.x | 全部 | 缓存/会话/锁 | 哨兵3节点 |
| 6 | Elasticsearch | 8.x | P2/P8 | 全文检索/日志 | 集群3节点 |
| 7 | ClickHouse | 23.x | P8 | 调用统计/分析 | 2节点+副本 |
| 8 | RabbitMQ | 3.12 | P1-P5/P8 | 消息队列 | 镜像2节点 |

**总计：8种引擎，~20个节点**

### 1.2 收敛评估

| 收敛方案 | 可行性 | 节省 | 风险 |
|---------|--------|------|------|
| MongoDB→MySQL JSON（P5日志） | ✅ 高 | -1种引擎 | MySQL JSON性能足够 |
| ClickHouse→Elasticsearch聚合 | ❌ 低 | -1种引擎 | ES聚合性能不如ClickHouse，不建议 |

**建议**：将P5的MongoDB改为MySQL JSON存储，减少至**7种**。已统一为MySQL作为唯一关系型数据库。

## 三、统一备份恢复方案

### 3.1 RPO/RTO统一定义

| 引擎 | 关联产品 | RPO | RTO | 备份策略 |
|------|---------|-----|-----|---------|
| MySQL | 全部业务线 | <1小时 | <30分钟 | binlog+每日全量+binlog point-in-time recovery |
| Oracle | P7（只读） | N/A | N/A | 由财务部门维护 |
| Milvus | P1-P4/P8 | <4小时 | <2小时 | 每日Collection快照+S3备份 |
| Redis | 全部 | <1小时 | <5分钟 | RDB每小时+AOF |
| Elasticsearch | P2/P8 | <4小时 | <2小时 | Snapshot Repository每日 |
| ClickHouse | P8 | <24小时 | <4小时 | 每日全量+增量分区备份 |
| RabbitMQ | P1-P5/P8 | <1小时 | <15分钟 | 定义+镜像队列 |

### 3.2 备份执行计划

| 时间 | 操作 | 引擎 | 保留策略 |
|------|------|------|---------|

| **每小时** | RDB快照 | Redis | 24个 |
| **每日凌晨2:00** | 全量备份 | MySQL | 30天 |
| **每日凌晨3:00** | Collection快照 | Milvus | 7天 |

| **每日凌晨4:00** | 分区备份 | ClickHouse | 30天 |

## 四、故障恢复SOP

### 4.1 故障分级

| 级别 | 定义 | 响应时间 | 处理人 |
|------|------|---------|--------|
| **P0 紧急** | 核心库不可用（MySQL主库/Redis/RabbitMQ） | <5分钟 | 后端开发+AI架构师 |
| **P1 严重** | 从库/搜索/向量库不可用 | <15分钟 | 后端开发 |
| **P2 一般** | 性能下降/慢查询/磁盘告警 | <1小时 | 后端开发 |
| **P3 低** | 日志/分析库异常 | <4小时 | 轮值人员 |

### 4.2 常见故障处理

| 故障 | 检测 | 自动处理 | 手动处理 |
|------|------|---------|---------|
| MySQL主库down | 心跳超时3次 | 从库自动切主 | 检查binlog完整性→恢复 |
| Redis内存溢出 | 内存>90% | 自动淘汰（allkeys-lru） | 排查大Key→清理 |
| Milvus搜索超时 | P99>1s持续5分钟 | — | 检查segment数量→compact |
| RabbitMQ队列堆积 | 深度>10000 | — | 增加消费者/扩容 |
| ES集群红色 | cluster_health=red | — | 检查未分配分片→修复 |
| ClickHouse合并卡死 | merge_queue>200 | — | 手动optimize partition |

## 五、自动化运维脚本

### 5.1 每日健康检查脚本（建议cron执行）

```bash
#!/bin/bash
# daily_db_health_check.sh

echo "=== 数据存储日检报告 $(date) ==="

# MySQL
echo "--- MySQL ---"
mysql -e "SHOW STATUS LIKE 'Threads_connected';"
mysql -e "SELECT table_schema AS db, ROUND(SUM(data_length + index_length) / 1024 / 1024, 2) AS 'size_mb' FROM information_schema.tables GROUP BY table_schema;"
mysql -e "SHOW SLAVE STATUS\G" | grep -E 'Seconds_Behind_Master|Slave_IO_Running|Slave_SQL_Running'

# Redis
echo "--- Redis ---"
redis-cli info memory | grep used_memory_human
redis-cli info keyspace
redis-cli info stats | grep keyspace_hits

# Milvus
echo "--- Milvus ---"
curl -s localhost:9091/api/v1/health

# RabbitMQ
echo "--- RabbitMQ ---"
rabbitmqctl list_queues name messages consumers | awk '$2 > 100'

# Elasticsearch
echo "--- Elasticsearch ---"
curl -s localhost:9200/_cluster/health?pretty | jq '.status,.number_of_nodes,.active_shards'

# 磁盘使用
echo "--- Disk Usage ---"
df -h /data/mysql /data/milvus /data/redis /data/elasticsearch /data/clickhouse 2>/dev/null
```
